Before the projects, let me frame the thesis. My hypotheses are that deep learning can improve diagnosis and treatment through feature capture, that radiomic features can be integrated with clinical data, that pretrained models in federated systems enable faster diagnosis, and that federated learning improves model efficacy without compromising data privacy. From these, the research objectives are: to develop MediNet — a user-friendly web tool for horizontal federated deep learning in medical environments; to facilitate model development on clinical data; and to implement transfer learning in federated environments.

Why Federated Learning?

The Problem

Clinical data cannot be centralised — GDPR, patient confidentiality and hospital regulations prevent pooling data across institutions.

The Solution: Federated Learning

Bring the model to the data, not data to the model. Each hospital trains locally and shares only model parameters — never patient data.

DP Protects — But What Happens at Scale?

Differential Privacy — each round is safe

Gaussian noise added to gradients. One round gives an adversary no useful information — privacy is mathematically guaranteed.

But privacy cost ε accumulates over rounds

Each participation costs ε privacy units. Run 100 rounds, 1000 rounds… the accumulated cost Σεᵢ grows without bound.

Model inversion becomes feasible

A determined adversary accumulating enough gradient updates can reconstruct patient-level data. The "safe per round" guarantee no longer holds globally.

Privacy budget — Hospital A
Round 1
B = 0.90
Round 10
B = 0.55
Round 50
B = 0.20
Round 100
B ≈ 0 🚨
When B → 0, accumulated gradients can be used for model inversion. Patient data is at risk.
v2 Budget Policy: each client has a fixed budget. When exhausted, they are excluded automatically.
This is the key insight that motivated v2. Standard DP protects each round, but privacy cost accumulates. An adversary who collects enough gradient updates — across many rounds — can start reversing the model to extract patient data. We needed to limit total participation, not just per-round privacy. That's the Budget Policy.

Visual Model Builder — Graph-Based (v2)

Any layer → any layer
Connect layers freely — skip connections, parallel branches, merge points.
Richer architectures
ResNet-style, multi-input fusion, ensemble splits — all without code.
Under active development

v2 replaces the linear chain with a proper directed acyclic graph. You can connect any layer to any downstream layer — which unlocks skip connections (ResNet-style), parallel branches that merge later, and multi-input architectures. The UI design is done; the backend graph compiler that validates the DAG and translates it to a Keras Functional API model is under active development.

The Key Innovation: Budget Policy (v2)

Model deployment → privacy configuration

1
When deploying a model, the admin configures the privacy budget ε for that model
2
The platform suggests ε by medical domain — e.g. oncology ε = 1.0 · cardiology ε = 2.0 · research ε = 8.0
3
Each training round deducts \(\varepsilon_i(t)\) — when B → 0, the client is excluded automatically
Budget update per round for client \(i\)
$$B_i(t) = B_i(t-1) - \varepsilon_i(t)$$
\(\varepsilon_i(t)\) — privacy cost of round \(t\)  ·  participates iff \(B_i(t) > 0\)

Client status in a round:

Hospital A
Budget remaining: 80% → Participates
Hospital B
Budget remaining: 35% → Limited participation
Hospital C
Budget remaining: 5% → Excluded from round
Medinet v2 introduces the Budget Policy. Each client has an assigned privacy budget. If a hospital has participated heavily and its budget nears the limit, the system automatically excludes it. This adapts participation to the real sensitivity of each dataset.

What is EPIFLARE?

Use case: Epigenomics

A geneticist has a model trained on DNA methylation data (EPIC arrays) and wants to apply it to new patients — without sending data outside the institution.

New role: Geneticist

Uploads patient methylation data (CSV), loads ONNX models, and gets semantic per-patient predictions — all within the institution.

1
Upload methylation CSV (samples × CpGs)
2
Load ONNX model + define required CpGs
3
Auto-matching — software filters CpGs for each patient
4
Configure semantic labels per output class
5
View per-patient predictions with confidence
EPIFLARE targets a different workflow. Instead of federated training, the geneticist already has a pre-trained model and wants to apply it to new patients within their own platform. ONNX is a standard interoperable format compatible with PyTorch, scikit-learn, etc.

Key insight: models declare the CpG features they need. The platform automatically intersects those with the patient's available CpGs — no manual preprocessing required by the geneticist.
This is the key technical detail: when a model is uploaded, the geneticist specifies which CpG sites the model was trained on. Then, when patient data arrives, the platform automatically finds those specific CpGs from each patient's EPIC array — which has 850,000 CpGs but the model may only need a few hundred. This automatic matching is transparent to the geneticist.

Why sensor hacking?
Commercial sensors do not allow easy data export outside their official apps. Juggluco is an open-source app that reads sensors via NFC/BT, giving us real-time data access.
The advantage
Patients already wear these sensors for their own health management — no additional hardware cost, no separate certified device needed.
Commercial sensors like the FreeStyle Libre don't allow easy data export. Juggluco is an open-source app that reverse-engineers the NFC protocol and lets us read the data in real time. We connect our app to Juggluco's API to get the patient's data.

Data Management

Data Management Plan

Current situation

No real patient data is managed today.
All experiments and validation use fully anonymous datasets.
Federated architecture ensures data never leaves the local institution — no centralised storage.
Differential Privacy applied at training — gradients are mathematically private before leaving the node.

Built for future clinical use

Platform designed so that when real patient data arrives, privacy protocols are already in place.
Anonymisation and access-control architecture under design — aligned with GDPR and hospital data governance requirements.
Future pilot studies will follow ethics committee approval and institutional data sharing agreements.
Currently we don't manage any real patient data. Everything we work with is fully anonymous — public datasets for federated learning experiments, and synthetic data for EPIFLARE and the glucose app. The platform is being built with privacy-by-design: data never leaves the local institution, and differential privacy is applied before any gradients are shared. When we move to real clinical data, the infrastructure for anonymisation, access control, and GDPR compliance will already be in place.
